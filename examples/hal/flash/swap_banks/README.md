# __Example: *hal_flash_swap_banks*__

**Example version:** 2.0.0

[![User Manual](doc/read_the-UM.svg)](https://dev.st.com/stm32cube-docs/examples/arch-v1/en/index.html "An offline version is also available in the Cube firmware package.")

How to configure, program internal flash memory bank 1 and bank 2 and how to swap between both of them using the FLASH HAL API.


## __1. Detailed scenario__

__Initialization phase__: At the beginning of the `main()` function, the `mx_system_init()` function is called to initialize the peripherals, the system clock, and the SysTick. Then LED status is initialized.

The application executes the following __example steps__:

__Step 1__: Initialization

- Initializes the Flash interface and user button.

- Unlocks Flash control and Option Bytes (OB) control registers to allow memory operations.

__Step 2__: Bank Swap Operation

- Waits for the user to press and release the button.

- After button release, performs a swap between Flash memory Bank 1 and Bank 2.

- The status LED indicates the currently active bank by blinking at a specific frequency:

  - Running from Bank 1: LED blinks at a slow frequency.
  - Running from Bank 2: LED blinks at a fast frequency.

__End of example__:

The application enters an endless loop, continuously blinking the status LED at the frequency corresponding to the active bank.

If you enable `USE_TRACE`, you can follow these execution steps in the terminal logs:

```text
[INFO] Step 1: Device initialization COMPLETED and the Flash access is unlocked.
[INFO] Step 2: Swap bank is activated. Running from bank2.
[INFO] Step 2: Swap bank is deactivated. Running from bank1.
```


## __2. Example configuration__

### __2.1. Bank Swapping Configuration__

Check and configure the Option Bytes:

- Connect device to *STM32CubeProgrammer*.

- Go to *Option bytes*.

- Select *User configuration*.

- Check/Uncheck *SWAP_BANK* option to configure the bank mapping.

__SWAP_BANK = 0__

- *Bank1* is mapped to the lower address

- *Bank2* is mapped to the higher address

__SWAP_BANK = 1__

- *Bank2* is mapped to the lower address

- *Bank1* is mapped to the higher address

### __2.2. IDE Projects Configuration__

This example includes two separate IDE projects: one targeting Bank 1 and the other targeting Bank 2.

For each project, define the macros EXAMPLE_WAIT_DELAY_ON_MS and EXAMPLE_WAIT_DELAY_OFF_MS in the IDE or build system to set the LED blink frequency.

### __2.3. Flashing the Binaries__

__2.3.1. Principle__

Each flash bank contains a separate binary image:

- Bank 1 holds the binary for the main application.

- Bank 2 holds the binary for the alternate application.

__2.3.2. Using STM32CubeProgrammer__

When you use STM32CubeProgrammer to program the device, it always flashes the binary to the memory region currently mapped to the base address.

If SWAP_BANK is enabled: The programmer writes the binary to Bank 2.
If SWAP_BANK is disabled: The programmer writes the binary to Bank 1.

Sequence:

1. Activate SWAP_BANK: in STM32CubeProgrammer, check the SWAP_BANK option (OB -> User configuration -> SWAP_BANK) to map Bank 2 to the base address.

2. Flash Bank 2: program the device with the Bank 2 binary file.

3. Deactivate SWAP_BANK: in STM32CubeProgrammer, uncheck the SWAP_BANK option to restore Bank 1 mapping.

4. Flash Bank 1: program the device with the Bank 1 binary file.

__2.3.3. Using the IDE and STM32CubeProgrammer__

This procedure assumes that the SWAP_BANK option byte is disabled (set to 0).

1. Flash Bank 2: load the binary file generated by the FLASH_SwapBanks_Bank2 sub-project into Bank2 at Bank2 address (for example 0x08100000) using STM32CubeProgrammer.

   Do not run this project yet.

2. Open the IDE.

3. Flash Bank 1: set the FLASH_SwapBanks_Bank1 sub-project as the active project in your IDE and run it. This will program Bank 1 at the Bank1 address.

## __4. Troubleshooting__

[![Troubleshooting](doc/debug_with-Troubleshooting.svg)](https://dev.st.com/stm32cube-docs/examples/arch-v1/en/debug/debug_toc.html "An offline version is also available in the Cube firmware package.")

Here are the points of attention for this specific example:

__Project Structure:__ This example requires two separate projects, each designed to run from a different flash memory bank.

__Bank Assignment:__ Each project must be programmed and executed in its respective flash memory bank (Bank 1 or Bank 2).

__Device Requirements:__ The target STM32 device must support dual-bank Flash memory and bank swapping functionality.

__Runtime issues:__ In case of issue, verify first that both banks are correctly programmed with their respective binaries before running the example.
